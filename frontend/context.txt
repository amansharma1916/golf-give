## PROJECT OVERVIEW

This is a full-stack subscription-based web application called the Golf Charity Subscription Platform.
It combines golf performance tracking, monthly prize draws, and charitable giving.
The platform is emotion-driven and modern — it must NOT look or feel like a traditional golf website.
Design leads with charitable impact, not sport.


## REPO ARCHITECTURE

Single Git repository with two root-level folders: frontend/ and backend/
These are two completely independent Node.js projects, each with their own package.json.
They communicate over HTTP — frontend calls backend REST API endpoints.
There are no shared folders between them at the repo root level.

golf-charity-platform/         <- repo root
  frontend/                    <- React 18 app (Vite + TypeScript)
  backend/                     <- Express.js API (TypeScript)
  .github/

## TECH STACK

### Frontend (frontend/)
- Framework:      React 18 + TypeScript (Vite)
- Styling:        Plain CSS + CSS Modules — NO Tailwind, styled-components allowed , framer motions allowed all other within react is allowed , NO Emotion
- Animations:     anything within react 
- Routing:        React Router v6
- Client state:   Zustand
- Server state:   TanStack Query (React Query)
- HTTP client:    Axios (calls backend API)
- Build tool:     Vite

### Backend (backend/)
- Framework:      Express.js + TypeScript
- Runtime:        Node.js 18+
- Database:       Supabase (Postgres) via supabase-js

- Payments:       Stripe — SKIPPED FOR DEMO (see PAYMENT DEMO MODE section)

- Validation:     Zod (all request bodies validated before any logic runs)

- Frontend:       Vercel (new account, not personal)
- Backend:        Railway or Render (new account, not personal)

## STRICT CODING RULES — APPLIES TO BOTH FRONTEND AND BACKEND

1.  TypeScript everywhere — all files must be .ts or .tsx, strict mode enabled in tsconfig
2.  Never use 'any' type — use unknown and narrow it, or define a proper interface
3.  Named exports only from utility and lib files — no default exports except React components
4.  All environment variables must be validated with zod at startup (env.ts in each project)
5.  Never log sensitive data (tokens, keys, passwords, PII) to the console
6.  All async operations must use async/await with try/catch — no unhandled promise rejections
7.  Input validation with zod on every API endpoint before any business logic runs


## STRICT CODING RULES — FRONTEND ONLY

1.  Plain CSS only — use CSS Modules (.module.css) for all component styles
2.  All design values (colors, spacing, typography, radius, shadows) must use CSS custom
    properties defined in src/styles/tokens.css — never hardcode hex or pixel values
3.  No inline styles unless the value is dynamically computed from JS (e.g. width from state)
4.  Every component folder must contain:
      ComponentName/
        index.tsx
        ComponentName.module.css
        ComponentName.types.ts
5.  Never call Supabase directly from the frontend — all data goes through the backend REST API
6.  Never store JWT tokens in localStorage — use memory (Zustand) only
7.  Use React Router <Link> for all internal navigation — never use <a href> for internal links
8.  All API calls must go through src/services/ — never call fetch/axios directly from components


## STRICT CODING RULES — BACKEND ONLY

1.  All route definitions must be in src/routes/ — one file per resource
2.  All business logic must be in src/services/ — controllers only handle req/res parsing
3.  All Supabase queries must go through src/lib/supabase.ts — never import supabase-js elsewhere
4.  Auth middleware must run on every protected route — never trust userId from request body
5.  Never expose Supabase service role key or Stripe secret key in any response
6.  All responses must follow a consistent shape:
      Success: { success: true, data: <payload> }
      Error:   { success: false, error: { message: string, code?: string } }
7.  HTTP status codes must be accurate — 200, 201, 400, 401, 403, 404, 409, 500

## API DESIGN

Base URL
  Development:  http://localhost:4000/api
  Production:   https://your-backend.railway.app/api

Frontend Axios instance (frontend/src/lib/axios.ts) sets baseURL from VITE_API_BASE_URL.
A request interceptor automatically attaches Authorization: Bearer <supabaseJwt> on every call.

### Auth
  POST   /api/auth/register          Create user profile row after Supabase signup
  GET    /api/auth/me                Return current user profile + subscription status

### Subscriptions
  GET    /api/subscriptions/status         Get current user's subscription row
  POST   /api/subscriptions/mock-checkout  Demo: instantly activate subscription
  POST   /api/subscriptions/mock-cancel    Demo: cancel subscription
  POST   /api/subscriptions/checkout       Real Stripe Checkout — stub, implement later
  POST   /api/webhooks/stripe              Stripe webhook — stub, implement later

### Scores
  GET    /api/scores                 Get user's scores (max 5, newest first)
  POST   /api/scores                 Add score (enforces 5-score rolling rule server-side)
  PUT    /api/scores/:id             Edit a score
  DELETE /api/scores/:id             Delete a score

### Draws
  GET    /api/draws                  List all published draws
  GET    /api/draws/upcoming         Next scheduled draw info
  GET    /api/draws/:id              Draw details + prize pool breakdown
  POST   /api/draws/simulate         Admin: preview draw without persisting
  POST   /api/draws/:id/publish      Admin: publish official draw results
  PUT    /api/draws/:id/configure    Admin: set draw type (random | algorithmic)

### Charities
  GET    /api/charities              List all active charities (search + filter)
  GET    /api/charities/featured     Featured charities for homepage
  GET    /api/charities/:id          Single charity profile
  POST   /api/charities              Admin: create charity
  PUT    /api/charities/:id          Admin: update charity
  DELETE /api/charities/:id          Admin: soft delete charity

### Winners
  GET    /api/winners/me             Current user's winnings history
  POST   /api/winners/:id/upload-proof    Upload verification screenshot
  GET    /api/winners                Admin: all winners with payout status
  POST   /api/winners/:id/verify     Admin: approve or reject proof
  POST   /api/winners/:id/mark-paid  Admin: mark payout as completed

### Admin
  GET    /api/admin/users            All users with subscription status
  PUT    /api/admin/users/:id        Edit user profile or scores
  GET    /api/admin/reports          Totals: users, prize pool, charity contributions, stats


## DATABASE SCHEMA (Supabase Postgres)

Table: users
  id uuid PK (references auth.users)
  email text NOT NULL
  full_name text
  is_admin boolean DEFAULT false
  created_at timestamptz DEFAULT now()

Table: subscriptions
  id uuid PK
  user_id uuid FK -> users.id
  stripe_customer_id text           -- 'demo_customer_<userId>' in demo mode
  stripe_subscription_id text       -- 'demo_sub_<userId>_<timestamp>' in demo mode
  plan text CHECK (plan IN ('monthly', 'yearly'))
  status text CHECK (status IN ('active', 'inactive', 'cancelled', 'lapsed'))
  current_period_end timestamptz
  created_at timestamptz DEFAULT now()

Table: scores
  id uuid PK
  user_id uuid FK -> users.id
  score integer CHECK (score >= 1 AND score <= 45)   -- Stableford format
  played_at date NOT NULL
  created_at timestamptz DEFAULT now()
  -- MAX 5 per user. Oldest deleted on insert of 6th. Enforced in scores.service.ts.

Table: charities
  id uuid PK
  name text NOT NULL
  description text
  image_url text
  website_url text
  is_featured boolean DEFAULT false
  is_active boolean DEFAULT true
  created_at timestamptz DEFAULT now()

Table: charity_contributions
  id uuid PK
  user_id uuid FK -> users.id
  charity_id uuid FK -> charities.id
  percentage integer DEFAULT 10 CHECK (percentage >= 10 AND percentage <= 100)
  created_at timestamptz DEFAULT now()

Table: draws
  id uuid PK
  month date NOT NULL
  status text CHECK (status IN ('pending', 'simulated', 'published'))
  draw_type text CHECK (draw_type IN ('random', 'algorithmic'))
  winning_numbers integer[]
  jackpot_amount numeric DEFAULT 0
  rolled_over_amount numeric DEFAULT 0
  created_at timestamptz DEFAULT now()

Table: draw_entries
  id uuid PK
  draw_id uuid FK -> draws.id
  user_id uuid FK -> users.id
  score_snapshot integer[]      -- copy of user's 5 scores at draw time
  match_count integer           -- 3, 4, or 5
  is_winner boolean DEFAULT false

Table: prize_pools
  id uuid PK
  draw_id uuid FK -> draws.id
  tier text CHECK (tier IN ('five_match', 'four_match', 'three_match'))
  pool_percentage numeric        -- 40, 35, 25
  total_pool numeric
  winner_count integer DEFAULT 0
  prize_per_winner numeric
  rolled_over boolean DEFAULT false

Table: payouts
  id uuid PK
  draw_entry_id uuid FK -> draw_entries.id
  user_id uuid FK -> users.id
  amount numeric
  status text CHECK (status IN ('pending', 'verified', 'paid'))
  proof_url text
  admin_note text
  created_at timestamptz DEFAULT now()


## USER ROLES & ROUTE PROTECTION

1. Public Visitor   — no auth. Routes: /, /charities, /pricing, /how-it-works
2. Subscriber       — authenticated + subscription.status = 'active'
                      Routes: /dashboard/*, all /api/* user endpoints
3. Admin            — authenticated + users.is_admin = true
                      Routes: /admin/*, all /api/admin/* endpoints

Frontend guards (in router/):
  <ProtectedRoute>       checks Zustand userStore for valid JWT → else /login
  <SubscriberRoute>      checks subscription status = 'active' → else /pricing
  <AdminRoute>           checks user.is_admin = true → else /dashboard

Backend middleware chain:
  Protected user routes: auth.middleware → subscription.middleware → controller
  Admin routes:          auth.middleware → admin.middleware → controller
  Public routes:         no middleware

Auth middleware decodes Supabase JWT from Authorization: Bearer <token>,
verifies with Supabase, attaches typed user to req.user.
Never trust a userId sent in the request body.

## SCORE ENTRY RULES

- Stableford format — integer 1 to 45 inclusive
- Each score must include a played_at date
- Maximum 5 scores stored per user at any time
- When a 6th score is added, the oldest (by played_at) is deleted automatically
- Rolling rule enforced in backend scores.service.ts — never frontend only
- Scores returned newest first (ORDER BY played_at DESC)


## DRAW ENGINE (backend/src/draw-engine/)

random.ts     — 5 unique random integers between 1 and 45
algorithm.ts  — weighted draw based on score frequency across all active subscribers
               (configurable: favour most frequent OR least frequent numbers)
simulate.ts   — run draw logic, return preview results, do NOT write to DB
prize.ts      — calculate prize pool from active subscriber count at draw time
               Split: five_match=40%, four_match=35%, three_match=25%
               No five_match winner → rolled_over_amount carries to next draw
               Multiple winners in same tier → prize split equally
               All functions must be pure — no DB calls, no side effects


## CHARITY SYSTEM

- User selects one charity at signup (stored in charity_contributions)
- Default contribution: 10% of subscription fee
- User can increase percentage up to 100%
- Independent donation option not tied to subscription
- Directory supports text search and filter
- Individual profile: description, image, upcoming golf day events
- Featured charities shown on homepage via GET /api/charities/featured


## WINNER VERIFICATION FLOW

1. Draw published → draw_entries created → payouts rows created (status = 'pending')
2. Winner receives email via Resend
3. Winner uploads screenshot → POST /api/winners/:id/upload-proof
   File stored in Supabase Storage bucket 'winner-proofs', URL saved in payouts.proof_url
4. Admin reviews → POST /api/winners/:id/verify { approved: true | false }
5. Approved → status = 'verified'
6. Admin marks payment sent → POST /api/winners/:id/mark-paid → status = 'paid'

## NOTES FOR COPILOT

  - Draw engine is the most critical part — keep all functions pure and fully unit tested
  - 5-score rolling rule must be enforced in backend scores.service.ts, not on the frontend
  - Auth middleware must verify the Supabase JWT on every single protected request
  - Subscription status must be re-checked per request via middleware — not cached in JWT
  - Jackpot rollover: store rolled_over_amount on the draw record, add it to next draw's
    five_match pool total at prize calculation time
  - Winner proof files go to Supabase Storage bucket 'winner-proofs' — store only the URL in DB
  - CORS in backend app.ts must whitelist FRONTEND_URL from env — never use wildcard *
  - frontend/src/lib/axios.ts must attach Supabase session token as Authorization: Bearer
    on every request via a request interceptor
